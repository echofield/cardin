# CARDIN LITE V1

## Position

Cardin Lite is not a separate company, app, or backend.

It is a lighter operating mode inside Cardin.

Full Cardin sells:
- return
- propagation
- summit
- season tension

Cardin Lite sells:
- immediate return
- weak-hour activation
- first-visit conversion
- reactivation before churn

The split is:
- `Cardin Lite` = make people come back quickly
- `Cardin` = make people come back, propagate, and culminate

## Product Rule

One scan, one immediate benefit, one next step.

Not:
- long onboarding
- too much profile capture
- invite flow on first scan
- season story upfront
- grand prize upfront

Lite must stay operational and obvious.

## Customer Identity

The card remains the account.

Lite still uses:
- QR entry
- minimal info capture
- card creation
- card code access

Minimal info on first scan:
- first name
- phone or email

That is enough.

Do not ask in the main Lite flow:
- birthday
- style preferences
- multiple profile fields
- referral fields
- complex consent branches

## Lite V1 Core Mechanics

### 1. First Visit Welcome

Goal:
- convert hesitation into first purchase
- create the card identity

Flow:
1. customer scans
2. enters minimal info
3. receives immediate offer
4. card is created
5. comeback logic starts

Works for:
- cafe
- restaurant
- beaute
- boutique

### 2. Dead-Hour Activation

Goal:
- monetize low-value time slots
- bring local traffic back during weak hours

Flow:
- offer is only valid in a narrow window
- merchant chooses the weak-hour rule
- Cardin surfaces a live offer with expiry

Works first for:
- cafe
- restaurant

### 3. Next-Visit Lock

Goal:
- create one concrete reason to come back soon

Flow:
- after a visit, the customer gets a timed return offer
- example: come back within 7 days
- expiry is part of the value

Works for:
- cafe
- restaurant
- beaute
- boutique

### 4. Lapsed-Client Reactivation

Goal:
- catch churn before the customer forgets the place

Flow:
- Cardin tracks rhythm break
- after a defined absence window, Cardin sends a comeback prompt
- offer is personal and time-bound

Works for:
- cafe
- restaurant
- beaute
- boutique

### 5. Instant Rebook

Goal:
- lock the next appointment before churn starts

Flow:
- appointment ends
- customer scans
- immediate rebook advantage appears
- merchant captures the next visit while the customer is still present

Works first for:
- beaute
- hair
- nails

## Domino In Lite

Domino can exist in Lite, but not as the main promise.

Rule:
- Domino is secondary
- return comes first
- propagation comes after activation

Lite Domino rule:
- unlock only after the card becomes active
- one invite only at first
- no grand prize language upfront
- no chain logic shown in the first onboarding path

This keeps Lite sharp.

If Domino appears too early, Lite stops being Lite.

## What Lite V1 Explicitly Does Not Sell

Do not lead with:
- season mechanics
- grand prize
- Diamond status
- summit storytelling
- behavioral mystery loops
- complex referral trees
- stock-based personalization

These belong to full Cardin or later Lite extensions.

## Sector Priority

### Priority 1
- Cafe
- Beaute

Why:
- fastest proof
- clean return logic
- high merchant comprehension

### Priority 2
- Restaurant

Why:
- strong use cases
- slightly more operational complexity

### Priority 3
- Boutique

Why:
- best ideas often require heavier logic
- stock, size, gifting, and duo logic add complexity too early

## Keep Now

Keep now in Lite V1:
- first visit welcome
- dead-hour activation
- next-visit lock
- lapsed-client trigger
- instant rebook
- secondary Domino after activation

## Postpone

Postpone to later:
- weather trigger
- size-match instant offer
- gift buyer flow
- couple mechanic
- social split as a lead mechanic
- social unlock
- ghost offer
- anti-offer
- asymmetrical rewards
- scarcity cascade
- visible tier logic as a sales promise

## Advanced Mechanics To Keep In Reserve

Two advanced mechanics are worth keeping for later internal use:

### Decaying Offer
- simple outside
- strong urgency
- easy to understand
- economically useful

### Temporal Identity
- based on visit rhythm, not profile claims
- useful for later personalization
- should stay mostly invisible to the customer

## Lite Merchant Promise

Merchant-facing promise:
- fill weak hours
- recover lost clients
- create one clear reason to come back
- keep the system simple to operate

This is not a loyalty program.
This is not a marketing SaaS.
This is a return instrument.

## Lite UX Rule

The customer should understand the flow in under 5 seconds.

The merchant should understand the value in under 30 seconds.

If a Lite screen needs explanation, it is too heavy.

## Suggested Merchant Mode Split

At commercial/product level:
- `Cardin Lite`
- `Saison complete`

Same infrastructure:
- same card identity
- same QR entry
- same dashboard shell
- same customer card route

Different logic packs.

## Implementation Sequence

### Phase 1
- document Lite mode in product language
- merchant mode selection in onboarding or sales flow
- define Lite rule pack in code/config

### Phase 2
- implement first-visit welcome
- implement dead-hour activation
- implement next-visit lock
- implement lapsed-client trigger
- implement instant rebook

### Phase 3
- add secondary Domino unlock after activation
- add basic merchant readout for return and reactivation

### Phase 4
- test decaying offer as first advanced extension

## Non-Negotiable Constraint

Lite must never become a messy shortcut to full Cardin.

If we add:
- too many fields
- too many mechanics
- too much explanation
- too much social logic too early

then we lose the point.

Lite wins only if it stays brutally clear.
